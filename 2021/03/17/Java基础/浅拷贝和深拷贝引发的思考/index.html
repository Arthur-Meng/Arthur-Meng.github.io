浅拷贝和深拷贝引发的思考

  众所周知，拷贝有浅拷贝和深拷贝。但是在研究过程中我发现了很多有意思的地方，我们先来看一个例子：

问题

  • super.clone()做了什么？
  • 为什么这样子拷贝算是浅拷贝？深拷贝该怎么做？
  • 重写clone方法为什么要实现Cloneable接口，明明Cloneable接口是个空的接口？
  • clone方法是Object类里面的方法，如果不实现Cloneable接口，会怎么样？

解答

问题一

  在上一篇文章中，我们讲到super是一个Java关键字，代表了当前对象中从父类继承过来的那部分特征，而所有类都继承了Object，所以最终super.clone()会调用到Object的clone()方法，而Object的clone()方法是一个native方法（jvm托管调用的本地方法），使用了RTTI（run-time type identification）的机制，动态得找到目前正在调用clone方法的那个reference，根据它的大小申请内存空间，然后进行bitwise的复制，将该对象的内存空间完全复制到新的空间中去，从而达到shallowcopy的目的。
  所以总而言之，super.clone()就是一模一样拷贝了一个当前的对象。

问题二

  在解释问题二前，我们先来看一个知识点，Java中的数据类型分为基本数据类型（六种数字类型（四个整数型，两个浮点型），一种字符类型，还有一种布尔型）和引用数据类型（除了基本数据类型的都是引用类型）。
  对于这两种数据类型，我们在进行赋值操作、用作方法参数或返回值时，会有值传递和引用（地址）传递的差别。在我们这个例子就是的体现是，对象里面的字段如果是基础数据类型， 比如int，那么存储的是值；如果字段是对象，那边存储的是引用。
  说到这里就解释了问题二，为什么这样子算是浅拷贝，因为我们是一模一样拷贝了一个对象，如果这个对象里面引用了别的，那这个引用也拷贝过去了，两个对象都指向了同一个引用。一旦这个引用发生变化，那这两个对象都会受到影响。那深拷贝该怎么做呢？答案就是把引用也进行拷贝，在上面的例子里面就是把a.b = b.clone();的注释给放开。这样子的话，新拷贝的对象里面会指向新的引用，那这两个对象就是完全独立的了。

问题三和四

  问题三和四可以合并说明。首先，Cloneable接口虽然没有任何内容，但是对于重写clone方法是必要的，原因是Cloneable接口是一个标记接口（类似于Serializable）。我们只有加了这个标记接口，才能触发RTTI机制（不清楚是编译的时候）。我们如果不实现Cloneable接口，那么在运行的时候就会报出是CloneNotSupportedException是，也就是我们在重写clone()方法的时候必须要显式抛出的报错（编译的时候不知道当前类或者继承的父类有没有实现Cloneable接口）。
