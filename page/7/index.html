StringUtils的使用

2020-04-29
编程

  这篇文章是我对之前学的StringUtils的整理，没什么技术含量，适用于刚刚毕业的崽，如果你已经是个中老手，就不用浪费时间往下看了。

介绍

  StringUtils是一个apache基金会提供的用于字符串处理的常用工具类，功能强大，使用方便，到几乎每个项目里面都用了，每个Java开发都或多或少见过和用过。熟练地掌握并使用它会让你在开发中事半功倍，代码可读性更高且更加健壮。

依赖


空字符串

  很多时候，我们在判断一个字符串是不是空字符串的时候，往往需要考虑很多情况，比如字符串对象是不是null，是不是空，是不是空格等。这会导致我们的判断写的很长，且容易遗漏某些判断。所以强烈建议使用StringUtils里面提供的对空字符串的判断的方法，简单明了。这些方法可以判断某一个字符串，也可以批量判断。常用的是判断某一个字符串的是不是空字符串的有以下4个方法：
1、public static boolean isEmpty(String str)：判断某字符串是否为空，为空的标准是为null或者长度为0

2、public static boolean isNotEmpty(String str)：判断某字符串是否不为空，不为空的标准是不为null且长度大于0（就是isEmpty()方法的反向判断）

3、public static boolean isBlank(String str)：判断某字符串是否为空，为空的标准是为null或者长度为0或者全是空白符组成

4、public static boolean isNotBlank(String str)：判断某字符串是否不为空，不为空的标准是不为null且长度大于0且不全是空白符组成（就是isBlank()方法的反向判断）

  我们可以发现isEmpty()和isBlank()不一样的地方是isBlank()更加严格，连空格等都算成是空的。个人感觉isBlank()和isNotBlank()会比isEmpty()和isNotEmpty()更建议用。

大小写

  Java原生的String类中就有一些对大小写的处理，与原生的不用的是，StringUtils兼容了对null的处理，当被处理的字符串为null时，会返回null（避免了空指针的报错）。

  StringUtils而且还拓展了更多的方法，首字母大写或者小写，判断字符串是不是全大写或者小写。

查找

  String原生也支持了indexOf的查找，不同的是，StringUtils兼容了对null的处理，除此之外还可以支持查询第几次出现的位置。

  • public static int indexOf(String str, char searchChar, int startPos)：返回字符 searchChar 从 startPos 开始在字符串 str 中第一次出现的位置。
  • public static int ordinalIndexOf(String str, String searchStr, int ordinal)：返回字符串 searchStr 在字符串 str 中第 ordinal 次出现的位置。

替换

  同样的，StringUtils的replace也兼容了对null的处理，除此之外还支持只会替换一次或者指定最大替换的个数。

移除

  还是同样的，StringUtils的remove也兼容了对null的处理，除此之外支持移除开头和结尾的（其实我们也可以用replace成空的字符串来完成remove的操作，但是个人不是很建议）。

截取

  截取也是我们常用的方法，StringUtils的截取支持指定位置，或者指定字符串前后截取。

反转

  反转是一个很有意思的功能，StringBuffer也有reverse方法，但是如果想直接处理还需要把String转为StringBuffer就比较麻烦，还是StringUtils的reverse比较简单。

  StringUtils的reverse还支持根据字符镜像反转。

其他

  commons-lang3包中， 除了StringUtils这个常用类之后，还有ArrayUtils（数组工具类），CharUtils（字符串工具类）,NumberUtils(数字工具类)等，大家有兴趣也可以去了解下！

前后端分离对接CAS的几种思路

2020-04-22
CAS

  前文中我们已经简单说了下项目怎么集成CAS客户端（如果没看过最好看下，不然可能下面的会看不懂），但是那些都是针对前后端不分离的系统的。而其实现在的很多项目，都是前后端分离的（这里的分离指的是部署分离），尤其是微服务架构的，很多甚至后台是网关。前端本身就已经像是一个服务了，它只是需要向后台请求数据。前后端分离时，我们的登录可能是这样子的：

  • 前端发现没登录，跳转到登录页
  • 用户在我们的登录页进行登录
  • 登录完成，生成会话信息（比如使用cookie或者session）
  • 正常访问

  那我们来想下如果我们按照之前的CAS登录思路来走应该要怎么样。我们首先来确认一点，前后端分离时，前端和后台谁来作为CAS客户端和CAS服务端交互？很显然，肯定是后端，前端显然不能胜任这一任务。那我们预想的登录可能是这样子的：

  • 前端发现没登录，这时候它该怎么办？它不能跳转到原先的登录页，而是去CAS的登录页
  • 到CAS的登录页进行登录的操作
  • 登录完成携带ST访问后台
  • 后台校验完ST之后再回到前端
  • 正常访问

  我们可以发现，整个流程和之前的对比最大的不同就是第一步。前端不能用自己原先那套登录逻辑了，而是发现没登录的时候跳转到CAS对应的登录页。那么我们应该怎么实现这一步？除此之外，我们需要考虑另外一个问题：如果另外的一个客户端登出，理论上我们的系统也要登出。那这时候我们前端请求的数据的时候，后台应该做出变动通知前端当前登录失效，需要重新登录。提供以下几个思路：

思路1

  前端发一个请求给后台，后台给出302，302的地址是CAS登录的地址。如果支持跨域的话，前端手动跳转到302的地址就可以了。但是因为CAS往往不是和前端部署在一起的，如果不做反向代理和CAS服务端不支持的话，基本上会发生跨域（不知道跨域的请自行百度）。如果为了避免跨域，我们可以把url就变成那个请求，这样子请求就不是AJAX请求，哪怕302也不算跨域。登出的时候也是这样子，前端发送登出请求，后台给出302，302的地址是CAS登出的地址。

思路2

  前端发一个请求给后台，后台给出200，响应里面是CAS登录的地址，然后前端跳转到CAS登录的地址。登出的时候也是这样子。这个思路和上面不一样就是把302变成了200，好处是前端可控，可以做更多自定义的操作。

思路3

  前端发现没登录，直接跳转到对应的CAS的登录页。登出的时候一样，直接把url变成对应的CAS登出请求。这种方法简单暴力，不需要和后台有交互，但是这样子的话，前端就需要配置CAS相关的信息（同时后台也是要配置的），就会和CAS产生耦合。

思路4

  我们可以增加一个适配器，这个适配器代理CAS服务端，适配器去支持跨域，然后整个流程和思路1就一样了，虽然有点麻烦，但是也是可行的（我就用过这种思路），用的是freemarker，稍微分享下代码：


思考

  所以我们仔细想下，会发现如果按照上面四个思路只有思路2和思路4是相对可行的。

  • 按照思路1，因为我们之后发的请求的肯定是ajax的，所以如果跨域，302就gg了。当然，如果你的CAS服务端支持跨域，那么思路1也是可行的，毕竟如果不发生跨域的话，前端就可以读取到302，并且手动跳转到302的地址。
  • 按照思路3，遇到另外的一个客户端登出的这种情况都不知道怎么办。
  • 所以只能按照思路2，在前端response的拦截器里面增加对应的处理，对数据增加判断，如果遇到后台说要跳转的信息，前端就跳转到响应里面的地址。
  • 或者思路4的方式，强行把跨域的CAS代理成不跨域的。

总结

  我们在前后端分离对接CAS时，会遇到一些平时没遇到的问题，这时候我们选用一个合适的方法见招拆招，去保证之前的CAS的流程可以顺利完成就可以了。只是这个合适的方法因项目而异，如果是CAS服务端支持跨域（一般很少），思路1无疑是最简单直接的。思路2可能是最容易让人接受的，而且也可以胜任绝大多数情况。思路3缺陷太大，无法兼容另一个客户端登出的情况。思路4则是比较麻烦，但是前端改动较少，只需要处理302的情况就可以了。

项目集成CAS客户端浅析

2020-04-20
CAS

  前文中我们已经简单说了下CAS是什么和CAS的服务端的改造，今天我们来简单说下CAS的客户端以及我们怎么在我们的项目里面集成CAS客户端。

集成CAS客户端后会怎么样

我们的项目集成了CAS客户端之后，项目内原先的登录体系就变成了CAS单点登录的体系。

原先我们的登录体系可能是这样子的：
1、访问我们的页面发现没有登录，重定向到我们的登录页
2、用户在我们的登录页进行登录
3、登录完成，生成会话信息（比如使用cookie或者session）
4、重定向到我们系统首页正常访问

变成了CAS单点登录的体系可能是这样子的：
1、访问我们的页面发现没有登录，重定向到CAS的登录页
2、用户在CAS服务端提供的登录页进行登录
3、登录完成后CAS服务端生成一个TGC票据（Ticket Grangting Ticket）和ST票据（Service Ticket）， TGT是CAS为用户签发的登录票据，有了TGT就表示用户在CAS服务端登录了， ST是为用户签发的访问某一客户端的票据。
4、CAS服务端会携带这个ST重定向到我们系统
5、我们系统向CAS服务端校验这个ST（如果不校验的话，就可以伪造这个ST来访问我们的系统）
6、校验完成之后，登录完成，生成会话信息
7、重定向到我们系统首页正常访问

如果登录了，也可能是这样子的：
1、访问我们的页面发现没有登录，重定向到CAS的登录页
2、重定向到CAS的登录页后，CAS发现这个当前已经有用户登录了（也就是cookie里面存了TGT）
3、CAS服务端生成一个ST票据
4、CAS服务端会携带这个ST重定向到我们系统
5、我们系统向CAS服务端校验这个ST
6、校验完成之后，登录完成，生成会话信息
7、重定向到我们系统首页正常访问

如果还是有点听不懂的话，我来打一个还算恰当的比方：
       比如我们的系统是摩天轮，我们付钱买了摩天轮的门票之后就可以玩摩天轮了。另外的系统可能是旋转木马什么的，我们玩旋转木马也需要付钱买旋转木马的门票。这样子我们每玩一个项目都要付钱买下票（输入用户名密码校验），就很麻烦。这就是我们之前的登录体系。
       所以有人就说我们就建一个游乐场，把所有的设施都放到一起，然后让我们去买游乐园的门票，有了游乐园门票我们就可以玩游乐园里面的所有设施了。这就是CAS单点登录的体系。这个比方在CAS已经登录场景中的映射是这样子的：我们去游乐园（服务端）玩，付钱（登录）买了游乐园的门票（TGT），有了门票我们就可以玩游乐园里面的设施了（表示已经登录了）。但是我们在玩摩天轮（客户端）的时候，摩天轮入口处的工作人员并不知道我们是不是已经买了票，所以会检查我们游乐园的门票（TGT），发现我们已经买了票我们就不用在付钱（再次登录），他们就会给我们一张摩天轮的门票（ST），我们拿着这个摩天轮的门票就可以玩摩天轮了！
       在CAS没有登录的场景中是这样子的：我们去玩摩天轮（客户端），摩天轮入口处的工作人员发现我们没有游乐园的门票（TGT），就让我们先去买游乐园的门票（TGT），等我们买了游乐园的门票之后他们再给我们一张摩天轮的门票（ST），我们拿着这个摩天轮的门票也可以玩摩天轮了！

集成CAS客户端需要配置什么

  我们可以很明显在上面的CAS登录的流程中发现，我们在配置客户端至少要配置服务端的在哪里（服务端的ip和端口），还要配置客户端自己在哪里（客户端的ip和端口，登录的时候需要通过url传递给服务端，服务端才可以在登录完成后携带ST准确重定向回到我们系统），并且还需要知道和服务端交互的接口。我们来整理下有哪些接口：

  • 登录的接口（我们发现没有登录，需要重定向到CAS的登录页）
  • 校验的接口（我们需要向CAS服务端校验登录完成后CAS服务端传给我们的ST）
  • 登出的接口（我们登出的时候需要请求CAS服务端，告诉它我们已经登出了，我们登录了之后其他的系统也需要登出了）

  不同的集成客户端的形式，这几项配置都不会少，但是配置会有所不同。如果你采用的较为高级的框架，可能只需要配置服务端信息和客户端信息，和服务端交互的接口已经默认配置好了，毕竟如果不魔改的话接口是不变的。基本上是这样子的：

接口 URL 协议
登录 /login
校验 /validate CAS 1.0
校验 /serviceValidate CAS 2.0
校验 /p3/serviceValidate CAS 3.0
登出 /logout

怎么集成CAS客户端

  我们在了解了整体流程和客户端需要配置什么之后，就可以对于正式开始的试验了。重要的事情说三遍：我们需要根据自己的项目的技术栈选择最合适的方式集成CAS客户端的方式，并且最好根据CAS服务端的版本来选择所集成客户端的版本！因为找到最合适的集成方式并且使用匹配的版本会让你在集成过程中少走很多弯路。
  由于集成CAS客户端的多样性和篇幅的限制，我就不一一具体讲解具体怎么集成，只是大概介绍下什么情况下应该使用说明方式集成。

非SpringBoot，非Shrio项目

1、集成cas-client，添加相应依赖（根据CAS客户端的版本来定）
2、配置web.xml，添加对应的CAS的过滤器filter，主要是以下这几个filter：

  • 拦截登录的filter（重定向到CAS的登录页）
  • 拦截校验ST请求的filter（接收到CAS服务端重定向到我们系统的携带了ST票据请求，向CAS服务端校验ST）
  • 拦截登出的filter（重定向到CAS的登出）

3、如果是前后端分离的话需要重写相关的filter，防止跨域失败
4、具体的流程和实现可以参考：传送门
5、也可以参照我之前随手写的demo：传送门（密码: defl ）

SpringBoot项目

1、集成cas-client-autoconfig-support
2、配置文件中加入cas相关的配置，配置很少，配置起来相当简单，舒胡
3、具体的流程和实现可以参考：传送门

Shiro项目

1、集成CAS Client时建议选用buji-pac4j而不是shiro-cas（shiro-cas不兼容高版本的CAS，已被弃用）
2、也是配置文件中加入cas相关的配置，但是由于需要和shiro有耦合，和之前的配置不大一样，需要多注意
3、具体的流程和实现可以参考：传送门上，传送门下

其他

  刚刚说的这几种都是传统的前后端不分离的系统，现在的很多新项目都是前后端分离。这时候我们在集成CAS客户端的时候会遇到一些其他的问题，如跨域等。我们下一章再讨论如果是前后端分离的话，我们在集成CAS客户端时需要注意什么和对应的几种思路。

从零改造CAS服务端

2020-04-15
CAS

CAS是什么

  CAS 是 Yale 大学发起的一个开源项目，旨在为 Web 应用系统提供一种可靠的单点登录方法。
  那么单点登录是什么？
  单点登录（Single Sign On），简称为 SSO，是目前比较流行的企业业务整合的解决方案之一。SSO的定义是在多个应用系统中，用户只需要登录一次就可以访问所有相互信任的应用系统。比如我们目前的系统存在诸多子系统，而这些子系统是分别部署在不同的服务器中，那么使用传统方式的session是无法解决的，我们需要使用相关的单点登录技术来解决。
  好吧，上面的都是套话，简单来说就是把多个不同的系统的登录捆成一个整体，一个登录了之后大家都登录了，一个登出了之后大家都登出了，统一登录（单点登录），统一登出（单点登出）。为了达到这个目标，我们必须要引入一个协调者，大家都去它那边执行登录和登出的操作。这个协调者就是CAS服务端。我们来简单想下这个协调者要做什么事情：

  • 如果一个系统没有登录，它就要向CAS服务端询问当前是不是已经有用户登录了，如果还没登录就要请求登录，CAS服务端需要提供登录页和登录的校验
  • 如果向CAS服务端询问时发现已经有用户登录，CAS服务端就要告诉这个系统其他的系统已经先登录了和当前登录用户是谁，它无需额外登录，可以直接生成当前用户登录的会话
  • 任意一个系统登出的时候，都要告诉CAS服务端当前用户登出了，CAS服务端要通知所有系统当前用户已经登出了，让各个子系统都把会话关闭

CAS登录流程

  当然以上只是总体的思路，具体实现的时候会稍微复杂一点。登录的流程大概是以下几步：
1、访问一个子系统发现没有登录，重定向到CAS服务端的登录页（携带service参数，参数的值是子系统的地址）
2、用户在CAS服务端提供的登录页进行登录
3、登录完成后CAS服务端生成一个TGC票据（Ticket Grangting Ticket）和ST票据（Service Ticket）， TGT是CAS为用户签发的登录票据，有了TGT就表示用户在CAS服务端登录了， ST是为用户签发的访问某一客户端的票据。
4、CAS服务端会携带这个ST重定向到子系统（重定向的地址就是登录的时候service的值，通过url传递ST，参数名是ticket）
5、子系统向CAS服务端校验这个ST（如果不校验的话，存在伪造这个ST来访问我们的系统的风险）
6、校验完成之后，登录完成，子系统生成会话信息
7、重定向到子系统页面正常访问

怎么移除https

  很多时候，我们自己在使用CAS服务端的时候（除非上生产），其实是并不需要https的。我们在使用https和证书之后反倒还会给客户端的接入造成影响（客户端后台访问我们的时候就会受限）。那我们怎么移除https呢？我们需要修改如下几个地方：

  • 修改WEB-INF/spring-configuration/warnCookieGenerator.xml文件，p:cookieSecure的值改为false

使用redis代替本地缓存（支持集群部署）

  我之前在查询CAS怎么支持集群部署的时候看到两种解决方案：一种是tomcat的共享session，一种是用分布式缓存代替本地缓存。共享session的话程序就不用改，但是每次部署都很麻烦，对工程有要求。如果是使用分布式缓存代替本地缓存，那我们就要重新里面的源码，有一定的代码量，但是部署起来方便。斟酌再三，我决定还是能用代码解决的就用代码解决，使用redis代替本地缓存。由于涉及到公司的机密，我就不贴代码了，但是稍微说下需要重写的类：

  • DefaultTicketRegistry：Ticket注册类，就是原先的TGT和ST的本地缓存，我们改写这个类里面的方法就可以了
  • CentralAuthenticationServiceImpl：统一校验服务，这里类里面需要改的是grantServiceTicket方法。当已经登录的情况下，第二个客户端再登录的时候，我们还额外需要更新redis里面的TGT（原先是本地缓存，拿出来了之后改了就可以了，不需要再放回去）

CAS的其他配置

  CAS服务端其实还有很多其他的配置（真的很多很多，毕竟是很成熟的项目），也可以针对不同的客户端做不同的配置（配置json的形式），我稍微挑几个讲下：

  • cas.properties的host.name：配置节点名
  • cas.properties的cas.logout.followServiceRedirects：配置登出之后是不是要跳转到登出url中传入的service地址
  • cas.properties的st.timeToKillInSeconds：ST校验有效时间（超时了ST就校验不过了）
