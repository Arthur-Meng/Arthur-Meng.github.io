分布式系统-缓存

2021-06-25
系统设计

  最近发现了一个很强的项目，介绍系统设计，感触颇深，所以想简单分享下分布式系统中可能用到的缓存。

作用

  缓存可以提高页面加载速度，并可以减少服务器和数据库的负载。在这个模型中，分发器先查看请求之前是否被响应过，如果有则将之前的结果直接返回，来省掉真正的处理。数据库分片均匀分布的读取是最好的。但是热门数据会让读取分布不均匀，这样就会造成瓶颈，如果在数据库前加个缓存，就会抹平不均匀的负载和突发流量对数据库的影响。


分类

客户端缓存

  缓存可以位于客户端（操作系统或者浏览器），服务端或者不同的缓存层。

CDN缓存

  CDN 也被视为一种缓存。

反向代理/Web服务器缓存

  反向代理可以直接提供静态和动态内容。Web服务器同样也可以缓存请求，返回相应结果而不必连接应用服务器。

应用缓存

  • 本地缓存：Guava、Caffeine
  • 分布式缓存：Ehcache、主流NoSQL(Redis，MemcacheDB，MongoDB)

数据库查询级别的缓存

  当你查询数据库的时候，将查询语句的哈希值与查询结果存储到缓存中。通常有ORM框架实现(如Mybatis)，估计会话不同会分成多几级缓存。

数据库缓存

  数据库的默认配置中通常包含缓存级别，针对一般用例进行了优化。调整配置，在不同情况下使用不同的模式可以进一步提高性能。


策略

Cache aside

  旁路缓存：操作逻辑是查询缓存，如果不存在那么就读取数据库并更新到缓存当中。如果是更新数据库，那么操作完数据库后，删除缓存（或者延时双删）。注意旁路缓存,缓存中的内容是不做更新操作的,只有写入和删除操作。适用于读操作较多的场景。而且实现很简单，性能也很好。

Read/Write through（直写模式）

  核心策略：以缓存操作为主，通过缓存读数据和写数据，由缓存与数据库交互。

  • Read Through:先查询缓存中数据是否存在,如果存在则直接返回,如果不存在,则由缓存组件负责从数据库中同步加载数据
  • Write Through:先查询要写入的数据在缓存中是否已经存在,如果已经存在,则更新缓存中的数据，并且由缓存组件同步更新到数据库中

  缓存与数据库保持一致，可能存在数据量过大的问题，而且写操作总是通过缓存到达主数据库，所以存在一定的写延迟。用于读操作较多的场景，但相较于Cache aside而言更适合缓存一致的场景（实现起来也更加麻烦）。

Write Back（回写模式）

  Write back是相较于Write Through而言的一种异步回写策略。异步写可以减少与物理磁盘存储的交互，也可以进行合并写等优化。实现比较复杂，可能会丢失数据，适用场于读少写多的场景。MySQL InnoDB 引擎的Cache Pool其实就是使用的WriteBack策略，相较于Write through 而言拥有更高的写入性能。

分布式系统设计所需基础知识

2021-06-25
系统设计

  最近发现了一个很强的项目，介绍系统设计，感触颇深，所以想简单分享下系统设计所需基础知识。

延迟与吞吐量

  • 延迟是执行操作或运算结果所花费的时间。
  • 吞吐量是单位时间内（执行）此类操作或运算的数量。
  • 通常，你应该以可接受级延迟下最大化吞吐量为目标。

CAP理论

  在一个分布式计算系统中，只能同时满足下列的两点:

  • 一致性（Consistency） ─ 每次访问都能获得最新数据但可能会收到错误响应
  • 可用性 （Availability）─ 每次访问都能收到非错响应，但不保证获取到最新数据
  • 分区容错性 （Partition tolerance）─ 在任意分区网络故障的情况下系统仍能继续运行

  由于网络并不可靠，所以我们应要支持分区容错性（总不能一个分区挂了之后整个都挂了吧？），并需要在软件可用性和一致性间做出取舍，所以我们有以下两个选择：

CP ─ 一致性和分区容错性

  等待分区节点的响应可能会导致延时错误。如果你的业务需求需要原子读写，CP是一个不错的选择。典型代表就是Zookeeper，一个选举的时间可能高达几十秒。

AP ─ 可用性与分区容错性

  响应节点上可用数据的最近版本可能并不是最新的。当分区解析完后，写入（操作）可能需要一些时间来传播。典型代表就是Eureka，Eureka各个节点都是平等的，几个节点挂掉不会影响正常节点的工作，只要有一台Eureka还在，就能保证注册服务可用，只不过查到的信息可能不是最新的。如果业务需求允许最终一致性，或当有外部故障时要求系统继续运行，AP是一个不错的选择。

为什么不能同时满足CA

  CA从本质需求上来讲就是矛盾的，如果我们在数据变动的时候，就需要分布式协商。而协商是需要时间的，在协商的过程中，如果每个节点继续提供服务，因为结果没出来，提供的数据肯定是不一致的，所以不满足一致性了；如果每个节点继续不提供服务，就不满足可用性了。


一致性模式

弱一致性

在写入之后，访问可能看到，也可能看不到（写入数据）。尽力优化之让其能访问最新数据。弱一致性在视频聊天和实时多人游戏等真实用例中表现不错。打个比方，如果你在通话中丢失信号几秒钟时间，当重新连接时你是听不到这几秒钟所说的话的。由于不需要考虑数据一致（丢失）的问题，性能也最好。

最终一致性

在写入后，访问最终能看到写入数据（通常在数毫秒内）。数据被异步复制。DNS和email等系统使用的是此种方式。最终一致性在高可用性系统中效果不错（很常见）。

强一致性

在写入后，访问立即可见。数据被同步复制。文件系统和关系型数据库（RDBMS）中使用的是此种方式。强一致性在需要记录的系统中运作良好，相对于来说性能也最差。


可用性模式

故障切换

单活：工作到备用切换（Active-passive）

关于工作到备用的故障切换流程是，工作服务器发送周期信号给待机中的备用服务器。如果周期信号中断，备用服务器切换成工作服务器的 IP 地址并恢复服务。宕机时间取决于备用服务器处于“热”待机状态（热备）还是需要从“冷”待机状态（冷备）进行启动。只有工作服务器处理流量。

工作到备用的故障切换也被称为主从切换。

双/多活：双工作切换（Active-active）

在双工作切换中，双方都在管控流量，在它们之间分散负载。如果有一方宕机了，流量就不从它那边走了。

故障切换的缺陷

  • 故障切换需要添加额外硬件并增加复杂性。
  • 如果新写入数据在能被复制到备用系统之前，工作系统出现了故障，则有可能会丢失数据（数据一致性问题）。

复制

  这里的复制主要是数据库部分的主─从复制和主─主复制。其实在我眼里，可用性的本质就是冗余，也就是复制，如果我们的系统存在不可用的情况，那就搞出很多个系统来，哪个能用用哪个。


域名系统

  域名系统是把www.example.com 等域名转换成 IP 地址。我们可以通过DNS服务来管理和路由流量:

加权轮询策略

  • 防止流量进入维护中的服务器
  • 在不同大小集群间负载均衡
  • A/B 测试

基于延迟路由（选择最低延迟）

基于地理位置路由（选择就近机房）


内容分发网络（CDN）

  内容分发网络（CDN）是一个全球性的代理服务器分布式网络，它从靠近用户的位置提供内容。通常，HTML/CSS/JS，图片和视频等静态内容由 CDN 提供。将内容存储在 CDN 上可以从两个方面来提供性能：从靠近用户的数据中心提供资源；通过 CDN 你的服务器不必真的处理请求。


负载均衡器

功能

  负载均衡器将传入的请求分发到应用服务器和数据库等计算资源。无论哪种情况，负载均衡器将从计算资源来的响应返回给恰当的客户端。负载均衡器的效用在于:

  • 防止请求进入不好的服务器
  • 防止资源过载
  • 帮助消除单一的故障点
  • SSL终结：解密传入的请求并加密服务器响应，后面就不需要用SSL了。
  • Session留存：追踪会话，同一个会话的请求路由到同一实例。

实现

  负载均衡器可以通过硬件（F5等）或 软件（HAProxy等）来实现，通常会设置采用工作─备用 或 双工作 模式的多个负载均衡器，以免发生故障。

负载策略

  • 随机
  • 最少负载
  • Session/cookie
  • 轮询调度或加权轮询调度算法
  • 四层负载均衡
  • 七层负载均衡

反向代理

介绍

  反向代理是一种可以集中地调用内部服务，并提供统一接口给外部。来自客户端的请求先被反向代理服务器转发到可响应请求的服务器，然后代理再把服务器的响应结果返回给客户端。

优点

  • 增加安全性：隐藏后端服务器的信息，屏蔽黑名单中的 IP，限制每个客户端的连接数。
  • 提高可扩展性和灵活性：客户端只能看到反向代理服务器的 IP，这使你可以增减服务器或者修改它们的配置。
  • SSL终结：解密传入的请求并加密服务器响应，后面就不需要用SSL了。
  • Session留存：追踪会话，同一个会话的请求路由到同一实例。
  • 压缩：压缩服务器响应
  • 缓存：直接返回命中的缓存结果
  • 静态内容：直接提供静态内容(HTML/CSS/JS/图片/视频等)

负载均衡器与反向代理

  反向代理有着负载均衡的作用，但是拓展了更多功能：压缩，缓存和静态内容

CAS会话过期方案浅析

2021-06-24
CAS

  最近有一次，突然被问到CAS服务端和CAS客户端的会话过期的一致性问题，有点懵逼。毕竟在我看来，既然都单点登录了，那边客户端的会话肯定是要和服务器的一致，客户端就没必要有自己的会话了，要过期就一起过期，还是太年轻了…

客户端会话过期

  我们试想一个问题，如果其中一个客户端非要有自己的会话过期策略，那应该怎么处理比较合理？把这个问题转换过来就是，如果某个客户端A自己过期了，那它有必要通知服务端过期吗？答案是否定的，毕竟如果通知了服务端过期，那服务端会通知别的客户端B之类的，B的用户万一在用着好好的，突然就被登出了，这不得骂人？
  那如果某个客户端A自己过期了，不通知服务端过期会有坏的影响吗？答案是有也没有。因为如果不通知服务端会话过期，那哪怕A自己把会话清了，下次重新访问的时候，还是会重定向到服务器重新获取一个票据，重新登录一次。好的方面是客户端A的这个过期没什么会影响…坏的方面是A的这个过期根本没什么用…OK，到这里还没结束，我们接着往下看。

服务端会话过期

  我们来试想另外一个场景，如果我们的需求是需要服务端的会话过期失效呢？我们应该怎么设计？
  抓瞎了呀…我的第一反应难道是心跳机制？客户端向服务端定时推送活跃会话，如果某个会话所有的客户端都不活跃，那就到期了把他踢出去。这样子好是好，但是太麻烦了，一点都不简洁。
  后期我查阅了一些资料，发现网上的关于会话过期都基本上都差不多，都是用的CAS原生的一些策略，基于AbstractCasExpirationPolicy类的实现有8个类，这些类都在cas-server-core-tickets的org.jasig.cas.ticket.support包下。这里主要介绍两种常用的：

  • NeverExpiresExpirationPolicy:票据永不过期
  • TicketGrantingTicketExpirationPolicy:票据具有固定的生存期和空闲使用超时时间（CAS默认的过期策略）

  其中关于TicketGrantingTicketExpirationPolicy网上很多文章真的很扯，描述统一都是当用户在2个小时（7200秒）之内不动移动鼠标或者进行系统超过8个小时（28800秒），则TGT过期。WTF？CAS服务端那么厉害？还能监控用户鼠标有没有移动？牛逼牛逼。于是乎我去看了下源码，发现完全不是那回事…

  后面又产生了疑问，那CAS服务端是怎么知道空闲了多久的？又是在什么时候去判断的有没有过期。看了好多，后来总算看到一篇稍微讲的合理的文章。TGT过期的校验发生在下一次登录的时候，即下一次登录，使用TGT的时候生成ST的时候都会校验是否有效，如果TGT已经过期那么整个流程就将失效，回到CAS单点登录的登录页面。讲到这里就一切跃然纸上了，TGT空闲时间就是上一次被登录的时候使用的时候到现在的间隔。校验发生在新登录发生的时候，如果TGT超过了最大存活时间或者最大空闲时间就算过期。
  这样子想来其实这个方案也还算合理。毕竟客户在登录新系统的时候肯定没在操作别的系统，这时候提示他超时了重新登录下也无可厚非。重新登录之后之前登录过的系统也会会再次生成新的会话，看上去似乎很完美！！！唯一的美中不足就是已经登录过的客户端在这种情况下其实是没有失效的情况的，但是我们可以结合上面的客户端会话来看，完美！！！我们可以给我们的客户端加上一个较小的会话过期时间，这样子如果会话失效了有两种情况：1、TGT没失效，自动重新登录，相当于没有过期；2、重新登录的时候校验TGT，发现也失效了，真的要手动重新登录。这样子就完美到达了我们的目的，既有会话失效，且两者的状态是统一的。

Semaphore

2021-05-25
多线程

总体介绍

  Semaphore用于限制可以访问某些资源（物理或逻辑的）的线程数目，他维护了一个许可证集合，有多少资源需要限制就维护多少许可证集合，假如这里有N个资源，那就对应于N个许可证，同一时刻也只能有N个线程访问。一个线程获取许可证就调用acquire方法，用完了释放资源就调用release方法。

主要方法

  • void acquire():从此信号量获取一个许可，在提供一个许可前一直将线程阻塞，否则线程被中断。
  • void release():释放一个许可，将其返回给信号量。
  • int availablePermits():返回此信号量中当前可用的许可数。
  • boolean hasQueuedThreads():查询是否有线程正在等待获取。

volatile关键字

2021-05-25
多线程

背景知识

  JVM中存在一个主内存(Java Heap Memory)，Java中所有变量都储存在主存中，对于所有线程都是共享的。每个线程都有自己的工作内存(Working Memory)。
  主内存主要包括本地方法区和堆。每个线程都有一个工作内存，工作内存中主要包括两个部分，一个是属于该线程私有的栈和对主存部分变量拷贝的寄存器(包括程序计数器PC和cup工作的高速缓存区)。所以工作内存中保存的是主存中某些变量的拷贝，线程对所有变量的操作都是在工作内存中进行，线程之间无法相互直接访问，变量传递均需要通过主存完成。

概要介绍

  与static一样，volatile是一个类型修饰符。volatile关键字解决了内存可见性（一个线程对共享变量值的修改,能够及时被其他线程看到）的问题，会使得所有对volatile变量的读写都会直接刷到主存，即保证了变量的可见性。
  volatile第二个作用是作为指令关键字，确保本条指令不会因编译器的优化而省略。即禁止指令重排优化。如果没有此关键字，JVM可以自由地进行一些优化，例如从不刷新某些线程中的本地副本，导致多线程的情况下发生不同步的问题，保证了执行时一定的”有序性”。

实现原理

  在JVM底层volatile是采用“内存屏障”来实现的。加入volatile关键字时，会多出一个lock前缀指令，lock前缀指令实际上相当于一个内存屏障（也成内存栅栏），内存屏障会提供3个功能：

  • 它确保指令重排序时不会把其后面的指令排到内存屏障之前的位置，也不会把前面的指令排到内存屏障的后面；即在执行到内存屏障这句指令时，在它前面的操作已经全部完成
  • 它会强制将对缓存的修改操作立即写入主存
  • 如果是写操作，它会导致其他CPU中对应的缓存行无效

volatile和synchronized的区别

  • volatile本质是在告诉jvm当前变量在寄存器（工作内存）中的值是不确定的，需要从主存中读取； synchronized则是锁定当前变量，只有当前线程可以访问该变量，其他线程被阻塞住。
  • volatile仅能使用在变量级别；synchronized则可以使用在变量、方法、和类级别的
  • volatile仅能实现变量的修改可见性，不能·保证原子性；而synchronized则可以保证变量的修改可见性和原子性
  • volatile不会造成线程的阻塞；synchronized可能会造成线程的阻塞
  • volatile标记的变量不会被编译器优化；synchronized标记的变量可以被编译器优化

volatile与static

  static保证了变量唯一性，static修饰符描述的变量表示在主存中所有此类的实例用的是同一份，各个线程创建时需要从主存同一个位置拷贝到自己工作内存中去(而不是拷贝此类不同实例中的这个变量的值)。也就是说只能保证线程创建时，变量的值是相同来源的，运行时还是使用各自工作内存中的值，依然会有不同步的问题。但是volatile保证了可见性，可以解决这个同步的问题。

SPI与JDBC

2021-04-07
编程

SPI是什么

  SPI的全名为Service Provider Interface。产生的原因是: 系统里抽象的各个模块，往往有很多不同的实现方案，在面向的对象的设计里，一般推荐模块之间基于接口编程，模块之间不对实现类进行硬编码。一旦代码里涉及具体的实现类，就违反了可拔插的原则，如果需要替换一种实现，就需要修改代码。为了实现在模块装配的时候能不在程序里动态指明，这就需要一种服务发现机制。 Vava的SPI就是提供这样的一个机制：为某个接口寻找服务实现的机制（解耦）。

SPI实现方案

  SPI的具体约定为:当服务的提供者，提供了服务接口的一种实现之后，在jar包的META-INF/services/目录里同时创建一个以服务接口命名的文件。该文件里就是实现该服务接口的具体实现类。而当外部程序装配这个模块的时候，就能通过该jar包META-INF/services/里的配置文件找到具体的实现类名，并装载实例化，完成模块的注入。
  基于这样一个约定就能很好的找到服务接口的实现类，而不需要再代码里制定。jdk提供服务实现查找的一个工具类：java.util.ServiceLoader。

SPI实现原理

  为了实现SPI，Java设计团队只好引入了一个不太优雅的设计：线程上下文类加载器（Thread Context ClassLoader）。这个类加载器可以通过java.lang.Thread类的setContextClassLoader方法进行设置。如果创建线程时还未设置，它将会从父线程中继承一个，如果在应用程序的全局范围内都没有设置过多的话，那这个类加载器默认即使应用程序类加载器。
  嘿嘿，有了线程上下文加载器，JNDI服务使用这个线程上下文加载器去加载所需要的SPI代码，也就是父类加载器请求子类加载器去完成类加载的动作，这种行为实际上就是打通了双亲委派模型的层次结构来逆向使用类加载器，实际上已经违背了双亲委派模型的一般性原则。但这无可奈何，Java中所有涉及SPI的加载动作基本胜都采用这种方式。例如JNDI，JDBC，JCE，JAXB，JBI等。

近代哲学备忘录

2021-03-23
哲学

  近代哲学备忘录是为了记下自己最近看过的哲学家及其思想，方便以后回顾和记忆，用于感(zhuang)悟(x)。

近代哲学

  15世纪中—19世纪40年代，称为近代哲学。

布鲁诺

  • 反教会、反经院哲学、反对地心说
  • 汲取了文艺复兴时期先进哲学和自然科学成果，论证了唯物主义和辩证法思想
  • 万事万物都处在普遍联系和不断运动变化，人类也历史是不断变化和前进的，所以主张社会变革

笛卡尔

  • 近代唯理论的第一个代表，近代西方哲学的三大理性主义者之一
  • 从普遍的怀疑开始的，把一切打倒，重新建立，完全运用思维以达到客观真理
  • 名言：我思故我在
  • 认为有神，精神，物质三个实体，所以是典型的二元论者：除了机械的世界外，还有一个精神世界
  • 提出“天赋观念”说

斯宾诺莎

  • 近代西方哲学的三大理性主义者之二
  • 一元论者或泛神论者（神无处不在，与世界一体）：实体只有一个，即是“自然即神化身”
  • 所以只要和上帝达成一致，人们就不再受制于这种影响，能获得相对的自由
  • 决定论者，他认为所有已发生事情的出现绝对贯穿着必然的作用
  • 经历：由于不承认神是自然的创造主，其认为自然本身就是神化身被教会驱逐，只能悲惨地磨镜片，因为吸入了大量的硒尘而死。

莱布尼茨

  • 近代西方哲学的三大理性主义者之三
  • 在数学史和哲学史上都占有重要地位（微积分）
  • 欧陆理性主义哲学的高峰：提出了单子论，单子独立和封闭，但是可以通过神彼此互相发生作用；可能性是在神的目光之下的实在，而没有真正的将世界的可能性看作为可能性
  • 名言：我们的宇宙，在某种意义上是上帝所创造的最好的一个

洛克

  • 唯物主义的经验论
  • 反对笛卡尔的“天赋观念”说，提出白板说：人的心灵开始时就像一块“白板”，而向它提供精神内容的是经验

培根

  • 实验科学的创始人，近代归纳法的创始人
  • 经验主义
  • 在哲学史上，培根的地位似乎微不足道

休谟

  • 彻底的怀疑主义（不可知论）
  • 休谟问题：从“是”能否推出“应该”，又称“实然与应然问题”
  • 对于道德问题，科学是无能为力的，科学只能回答“是什么”的问题，而不能告诉我们“应该怎样”的问题

康德

  • 提出了三大批判，即《纯粹理性批判》、《实践理性批判》和《判断力批判》
  • 《纯粹理性批判》回答了：我们能知道什么？真理似乎只有在人的头脑中才成为可能，即知识的普遍必然性，调和了勒内·笛卡儿的理性主义与弗朗西斯·培根的经验主义：他一方面赞同经验论的观点，即世人的一切知识都是经验知识，离开经验世人没有任何知识；另一方面，也赞同唯理论对经验论的批评：对科学知识来说仅有经验是不够的，它们的普遍必然性只能是先天的
  • 《实践理性批判》，对于实践理性的“批判”就是要考察那规定道德行为的“意志”的本质以及它们遵循的原则，提出了人是目的而非手段
  • 康德给哲学带来了巨大转变：不是事物在影响人，而是人在影响事物；是人在构造现实世界，在认识事物的过程中，人比事物本身更重要
  • 提出了著名论断就是：人是万物的尺度；人其实根本不可能认识到事物的真性，我们只能认识事物的表象。

黑格尔

  • 古代辩证法：在辩论中以揭露对方议论中的矛盾并克服这些矛盾来求得真理的方法
  • 黑格尔的辩证法认为整个历史和精神的世界是一个过程，就是说是在不断地运动着，变化着，发展着和改造着的，过程内部的矛盾是事物自身运动和发展的源泉
  • 辩证法是由正题、反题与合题组成的，即三段式解读法：完全按照太极图的正、反、和的三维形式
  • 强调逻辑：逻辑学是这个世界的“理性结构”——也是我们精神认识这个世界的逻辑
  • 客观唯心主义：绝对精神是世界的本源，世界的运动变化乃是“绝对精神”自我发展的结果
  • 名言：存在即合理（准确是翻译是凡合乎理性的即是真实的，凡真实的即是合乎理性的；即”绝对精神“自在地具备着一切，然后外化出其他，最后在更高的层次上回归自身。因此，凡是在这个发展轨迹上的就是合理的）

马克思

  • 马克思通过黑格尔创建的辩证法创建了辩证唯物主义和历史唯物主义
  • 辩证唯物主义：物质是第一性的，意识是第二性的；事物的矛盾规律，即对立统一的规律，它是物质世界运动、变化和发展的最根本的规律
  • 历史唯物主义：社会历史的发展有其自身固有的客观规律
