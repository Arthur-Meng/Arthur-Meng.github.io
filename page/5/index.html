栅栏和闭锁

2020-12-12
多线程

  在并发编程里面，栅栏和闭锁是两个常用的工具类，本文就带大（zi）家（ji）来一睹他们的真容（小白向）。

闭锁（CountDownLatch）

  闭锁（CountDownLatch）允许一个或多个线程，等待其他一组线程完成操作后，再继续执行。打个比方，我在并发的时候需要控制执行顺序，需要在某些线程之后完成后才能继续执行下面的步骤，这时候引入闭锁就可以很好的解决这个问题。实现方式如下：

  • 闭锁通过一个计数器来实现的，我们定义闭锁的时候需要定义计数器的初始值，这个值就是需要等待线程的数量。
  • 我们在需要把闭锁传给需要等待的线程，并且在等待的地方调用CountDownLatch的await()方法。
  • 当需要等待的线程完成了自己的任务后，调用CountDownLatch的countDown()方法，计数器的值就会减1。
  • 当计数器值为0时，说明所有的等待线程都执行完了，此时，在闭锁上等待的线程await()就结束了，就可以恢复执行任务。

栅栏（CyclicBarrier）

  栅栏（CyclicBarrier）类似于闭锁，它也能够阻塞一组线程直到某个事件发生；但是它与闭锁(CountDownLatch)的重大区别是，闭锁是所有线程等待一个外部事件的发生；而栅栏则是所有线程相互等待，直到所有线程都到达某一点时才打开栅栏，然后线程可以继续执行。打一个比方，栅栏就是赛马的比赛时候在马前边的栏杆，只有等全部赛马都上了赛道准备好了之后，栏杆才会打开，比赛才会开始。

动态规划的理解和应用

2020-12-02
算法

  珠玉在前,瓦石难当。如果大家真的想好好了解下动态规划，请移步labuladong的算法小抄。里面讲解的深入浅出，受益匪浅。本文仅仅是记录一些个人在使用过程中的感悟。

核心

  私以为动态规划的核心就想清楚怎么处理子问题：在子问题中，我们需要什么数据，需要做什么处理才能过渡到下一个子问题。就labuladong的算法小抄里面的话说就是找到状态转移方程：怎么根据现有数据转移到下一个状态。所以其实还是那两个问题，我们在子问题中需要什么样的数据支撑我们进行当前的处理（这个数据肯定是和状态息息相关的），怎么处理当前的数据过渡到下一个子问题。

使用方法

Table Fill

  Table Fill其实就是填dp表（Dynamic Programming，DP），dp表就上面说的支撑我们进行处理的额外数据（我们每次处理的结果都会存到dp表，下一个处理的时候从dp表取数据处理）。dp表往往是由你后续处理决定的，后续处理时需要这个额外数据，那它就应该存在dp表。dp表可以是一个，可以是多个，可以是一维，也可以是多维，它的形式取决于你需要什么样的数据。
  我们在填dp表时，需要注意的是遍历顺序。我们可以从前往后填，也可以从后往前填，亦或是从左或者从右，甚至可以斜着来。遍历顺序取决于你的场景，什么顺序遍历所需要的计算越小就采用什么顺序。
  Table Fill大概率是动态规划的最优解，尤其是那种统计的题目。但是难点依旧在找到合适的状态转移方程（有一些状态转移方程甚至可能很晦涩，看不懂）。

Recursion+Memoization

  Recursion+Memoization就是递归+备忘录。相较于Table Fill，递归+备忘录的难度直线下降，写出来的代码都是看得懂的。如果你在做算法题的时候，遇到了一道你觉得很操蛋的，用正常思维解决不了的题目，那么你基本上可以往动态规划靠，而动态规划最简单的实现就递归+备忘录。一旦你找到想好递归中如何处理子系统和想好边界的判断，那么这道题目就基本上解出来了。
  其实这两个方法都是动态规划，原理其实是一样的，只是实现方法不一样，所以递归+备忘录的形式基本上可以转换成Table Fill的形式。
  递归+备忘录的性能其实可以接受的（如果没有备忘录就接受不了了），但是缺点是耗费堆栈，毕竟在递归的时候会频繁创建栈去执行子问题，而Table Fill只是在一个方法内循环。所以实际情况下还是更建议使用Table Fill。

线程池浅析

2020-12-01
多线程

  众所周知，多线程可以极大提高执行效率。阿里的Java规范规定线程资源必须通过线程池提供，不允许在应用中自行显式创建线程。因为线程池的好处是减少在创建和销毁线程上所消耗的时间以及系统资源的开销，解决资源不足的问题。如果不使用线程池，有可能造成系统创建大量同类线程而导致消耗完内存或者”过度切换”的问题。

Executors

  Java通过Executors提供四种预设线程池，分别为：

  • newCachedThreadPool：创建一个可缓存线程池(线程池为无限大)，如果线程池长度超过处理需要，可灵活回收空闲线程，若无可回收，则新建线程。
  • newFixedThreadPool：创建一个定长线程池，可控制线程最大并发数，超出的线程会在队列中等待（这个感觉是比较常用的线程池）。
  • newScheduledThreadPool：创建一个定长线程池，支持定时及周期性任务执行。
  • newSingleThreadExecutor：创建一个单线程化的线程池，它只会用唯一的工作线程来执行任务，保证所有任务按照指定顺序(FIFO, LIFO, 优先级)执行。

ThreadPoolExecutor

  但是阿里Java规范规定线程池不允许使用Executors去创建，而是通过ThreadPoolExecutor的方式，这样的处理方式让写的同学更加明确线程池的运行规则，规避资源耗尽的风险。

  例如Executors预设的线程池对象可能存在如下弊端：

  • FixedThreadPool和SingleThreadPool：允许的请求队列长度为Integer.MAX_VALUE，可能会堆积大量的请求，从而导致OOM。
  • CachedThreadPool：允许的创建线程数量为Integer.MAX_VALUE，可能会创建大量的线程，从而导致OOM。

ThreadPoolExecutor使用参数

  释义如下：

序号 名称 类型 含义
1 corePoolSize int 核心线程池大小
2 maximumPoolSize int 最大线程池大小
3 keepAliveTime long 线程最大空闲时间
4 unit TimeUnit 时间单位
5 workQueue BlockingQueue 线程等待队列
6 threadFactory ThreadFactory 线程创建工厂
7 handler RejectedExecutionHandler 拒绝策略

常用加密算法介绍

2020-12-01
安全

  在我们的程序中，处于安全性的考虑，往往会使用加密算法对关键数据进行加密，所以熟悉一些常用的加密算法是非常有必要的事情。这可以帮助我们在对应的场景下选择最合适的加密算法。我们常用的加密算法有以下三种类型：对称加密、非对称加密和签名算法。下文一一展开他们分别是什么和适用于什么场景。

对称加密

  对称加密的特点是使用同一个密钥进行加密和解密，又称传统加密算法。加密解密过程：明文->密钥加密->密文，密文->密钥解密->明文。

  • 优点：计算量小，加密比非对称加密算法快，加密效率高。
  • 缺点：使用同一个秘钥，不够安全。

  所以对称加密适用于那种无需进行密钥交换的场景（如内部系统，事先就可以直接确定密钥）。并且由于加解密速度快，也适合数据内容比较大的加密场景。常用的有如下三种：

DES

  DES(Data Encryption Standard)：数据加密标准，但是现在用的很少，因为加密强度不够，能够暴力破解。

3DES

  3DES原理和DES几乎是一样的，但是使用3个密钥，对相同的数据执行三次加密，增强了加密强度。缺点是要维护3个密钥。

AES

  AES(Advanced Encryption Standard)：高级加密标准，目前美国联邦政府使用的就是AES加密，用于取代DES加密，是对称密钥加密中最流行的算法。如果采用真正的128位加密技术甚至256位加密技术，暴力破解要需要耗费相当长的时间。


非对称加密

  非对称加密算法又叫公开密钥算法。采用的是公钥和私钥相结合的加密方法。公钥和私钥是两个完全不同的密钥，一个用于加密，一个用于解密。同时这两个密钥在数学上是关联的。即解密算法不是加密算法的逆算法，因此在知道了加密算法后也无法知道解密算法，保证了安全性。

  • 优点：保密性比较好，消除了用户交换秘钥的需要，而且算法强度复杂，安全性比较强。
  • 缺点：速度慢于对称加密算法。

  所以适用于需要密钥交换的场景（如互联网应用，无法事先约定密钥）。而且与对称加密算法结合。利用非对称加密算法安全性较好的特点，传递对称加密算法的密钥。利用对称加密算法加解密速度快的特点，进行数据内容比较大的加密场景的加密。如HTTPS。常用的有如下：

RSA

  RSA算法的安全性基于RSA问题的困难性，也就是基于对极大整数做因数分解的难度决定了RSA算法的可靠性，所以RSA算法的保密强度随其密钥的长度增加而增强。但是密钥越长，其加解密所耗用的时间也越长。因此要根据所保护信息的敏感程度与攻击者破解所要花费的代价值不值得以及系统所要求的反应时间来综合考虑。


签名算法

  签名算法又称哈希算法、散列算法，是一种单向算法，它通过对数据内容进行散列得到一个固定长度的密文信息（信息是任意长度，而摘要是定长）。即用户可以通过哈希算法对目标信息生成一段特定长度的唯一的Hash值，却不能通过这个Hash值重新获得目标信息。该算法不可逆。

  • 优点：摘要是定长的，且摘要是不可逆的

  所以签名算法适用的场景很多，如对文件或者数据进行签名，确保信息传输完整一致，可以判断是否是同一个文件或者文件是否被篡改（这就是百度云盘有时候秒传大文件的原因）；对密码签名，确保除了用户之外没有人知道密码是什么。

MD5

  MD5即Message-Digest Algorithm 5（信息-摘要算法 5）是计算机广泛使用的散列算法之一，MD5的前身有MD2、MD3和MD4。MD5一度被广泛应用于安全领域，但是由于MD5的弱点被不断发现以及计算机能力不断的提升，现在已经可以构造两个具有相同MD5的信息，使MD5算法不再适合当前的安全环境。目前，MD5计算广泛应用于错误检查，例如：在一些BitTorrent下载中，软件通过计算MD5和检验下载到的碎片的完整性。


国密

  国密即国家密码局认定的国产密码算法。主要有SM1，SM2，SM3，SM4。上文所说的都是国际算法，由于安全问题，有些项目可能会强制要求使用国密算法，所以我们也需要了解。

SM1

  SM1为对称加密。其加密强度与AES相当，其加密强度与AES相当。该算法不公开，调用该算法时，需要通过加密芯片的接口进行调用。基于该算法已经研制了系列芯片、IC卡、智能密码钥匙、加密卡等安全产品。

SM2

  SM2为非对称加密，对标的是RSA。该算法已公开。由于该算法基于ECC（是什么请自行百度），故其签名速度与秘钥生成速度都快于RSA。

SM3

  SM3为摘要算法，对标的是MD5。算法已经公开，安全性相对较高，网上可以很轻松搜索到源码。其校验结果为256位。

SM4

  SM4为无线局域网标准的分组数据算法，也是对称加密，对标的是AES。密钥长度和分组长度均为128位。

PS：由于SM1、SM4加解密的分组大小为128bit，故对消息进行加解密时，若消息长度过长，需要进行分组，要消息长度不足，则要进行填充。

回溯算法的经典应用

2020-11-18
算法

  由于兴趣问题，本人最近在刷LeetCode，总结发现算法中最常出现的思想无非是动态规划，回溯算法，贪心算法和位图算法等。其中回溯算法出现的个人体感频次高居第二，所以记录一个回溯算法的经典应用：解数独。

怎么解

  首先我们通过程序解数独的总体思路还是和人解数独是一样的，就是在空白处填入当前可能的值直到全部值都被填完。
  但是解决方法可能截然不同，人做数独时（至少我是这样子的）会纵观整个数独，然后把那种根据当前条件可以确定的值填入，每填一个可以依靠的信息也越多，一个一个填下去直到全部填完，只有当迫不得已的时候才会在二个可能性里面二选一填入验证填入的值是不是对的。
  而程序做不到像人一样纵观全局（或者说很难，需要抽离出解数独的多种规并则转换为代码），只能在当前的可能性里面一个一个试过去，比如当前的空可以填1，2，3三个值。程序会假设填入1，然后递归下去当当前空为1的时候是否可能存在解，如果不存在，再填入2，3。我们在每一个递归里面需要做的，就是在找到当前空白处的所有可能性并递归下去。如果全部填完则返回成功，如果找不到值填入或者所有可能的值填入后都失败则返回失败。可以试想下，这段递归的代码最核心的代码是两块：怎么快速找到空白处的所有可能性和怎么用最低的代价递归下去（尝试之后的可能性）。

如何运用回溯算法

  回溯算法就是为了解决上述的第二个问题：怎么用最低的代价递归下去。我们在没有用回溯思想的时候可能会这样子做，深拷贝当前的数独，然后将可能的值填入之后再递归下去。这样子做从逻辑上来说是没什么问题，但是太吃内存了，每一次递归都会产生一个新的数独对象。而且由于大量创建对象，处理速度也很慢。
  这时候我们是回溯算法就闪亮登场了，由于我们是在递归，我们无需拷贝当前数独，直接将可能的值填入之后再递归下去，如果成功最好，如果失败了再把状态还原，就相当于这次尝试失效，一切还原，我们来进行下一个尝试。有木有，有木有，这个感觉很像是原地算法，我们只需要用当前的一个数独对象就可以遍历所有结果。还是之前那个例子，比如当前的空可以填1，2，3三个值。程序会假设填入1，递归下去如果1找不出解，则还原状态，当前数独和尝试1之前是一样的，再尝试2和3两个值。
  所以回溯算法的核心思想就是在尝试的过程中寻找解，当发现已经求不出解时，就回溯返回（还原状态），尝试下一个可能性，直到找到解或者遍历完全部的可能性。
