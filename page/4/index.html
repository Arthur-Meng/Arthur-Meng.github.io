ThreadLocal妙用

2020-07-27
多线程

什么是ThreadLocal

  ThreadLoal变量是线程局部变量，同一个ThreadLocal所包含的对象，在不同的Thread中有不同的副本。这里有几点需要注意：

  • ThreadLocal提供了线程本地的实例。它与普通变量的区别在于，每个使用该变量的线程都会初始化一个完全独立的实例副本。
  • 因为每个Thread内有自己的实例副本，且该副本只能由当前Thread使用。这是也是ThreadLocal命名的由来。
  • 既然每个Thread有自己的实例副本，且其它Thread不可访问，那就不存在多线程间共享的问题。
  • ThreadLocal变量通常被private static修饰。当一个线程结束时，它所使用的所有ThreadLocal相对的实例副本都可被回收。
  • ThreadLocal是一个泛型类，保证可以接受任何类型的对象。

原理

  一个线程内可以存在多个ThreadLocal对象，所以其实是ThreadLocal内部维护了一个 Map ，这个Map不是直接使用的HashMap ，而是ThreadLoca 实现的一个叫做ThreadLocalMap的静态内部类。而我们使用的 get()、set() 方法其实都是调用了这个ThreadLocalMap类对应的 get()、set() 方法。

应用

  ThreadLoal提供了线程级别的信息存储，可以用于存储当前线程的重要标准和信息，并且可以全局共享，使得代码耦合度更低，且实现更优雅（尤其是和注解联合起来使用），例如：

  • 在接入层或者拦截器中将当前用户编号存入ThreadLoal，在之后的处理过程中都可以拿到当前用户编号
  • 通过注解将需要权限控制的标志存入ThreadLoal，在之后Mybatis的拦截器中根据标志判断是否需要进行权限过滤
  • 通过注解数据源信息存入ThreadLoal，在获取数据库连接时根据ThreadLoal中的数据源信息获取对应的连接

AOP

介绍

  我们在项目中可能会遇到这样子的困扰，有一些通用的业务逻辑写的到处都是，导致项目内存在大量重复的代码，不仅不美观而且而且还容易出错。这时候我们就可以使用AOP把代码中的通用部分抽取出来，在单独的地方统一处理。一些有效的尝试比如：

  • 日志记录
  • 异常处理
  • 权限校验
  • 多数据源
  • 数据库事务

  • 使用execution的匹配包下的所有方法

增强/通知（Advice）

  增强又称通知(Advice)，指代植入切面的事件，增强（通知）有以下几种类型：

  • 前置（before)：在方法执行之前执行通知
  • 后置（after)：在方法执行后通知，无论其结果如何
  • 后置返回（after-returning)：只有方法成功完成后才能在方法执行和执行通知
  • 后置异常（after-throwing)：只有方法抛出异常而退出方法执行后才能运行通知
  • 环绕（around)：在调用通知方法之前和之后运行通知

@Target

  @Target说明了Annotation所修饰的对象范围：Annotation可被用于 packages、types（类、接口、枚举、Annotation类型）、类型成员（方法、构造方法、成员变量、枚举值）、方法参数和本地变量（如循环变量、catch参数）。一般常用的是就是ElementType.TYPE，ElementType.FIELD，ElementType.METHOD和ElementType.PARAMETER。具体的枚举如下：

@Retention

  Retention注解的值是RetentionPolicy类型的，RetentionPolicy也是一个枚举类型，这个枚举决定了Retention注解应该如何去保持，RetentionPolicy有3个值：CLASS、RUNTIME 、SOURCE，按生命周期来划分可分为3类：

  • RetentionPolicy.SOURCE：注解只保留在源文件，当Java文件编译成class文件的时候，注解被遗弃；
  • RetentionPolicy.CLASS：注解被保留到class文件，但jvm加载class文件时候被遗弃，这是默认的生命周期；
  • RetentionPolicy.RUNTIME：注解不仅被保存到class文件中，jvm加载class文件之后，仍然存在；

  所以我们一般使用的是RetentionPolicy.RUNTIME。

@Documented

  @Documented注解表明这个注解是由javadoc记录的，在默认情况下也有类似的记录工具。如果一个类型声明被注解了文档化，它的注解成为公共API的一部分。

新项目应该如何架构

2020-06-05
编程

  最近由于工作的变动，会涉及到从零开发一个新项目，所以想简单分析下自己在构建新项目中的一些想法。

关于框架

  没什么好说的，如果没什么特殊情况的话，基本上是基于SpringBoot构建，围绕SpringBoot的体系选用其他的框架（当然如果公司自己有封装自己的框架，其实更加建议用公司封装的，毕竟万一到时候要整合什么其他的就麻烦了）。推荐使用SpringBoot初始化构建（如官网构建或者IDEA构建），直接选择需要集成到项目里面的框架和依赖。至于具体的选型，比如微服务是选dubbo还是SpringCloud全家桶，就不展开了，仁者见仁。


关于模块和项目构建

  • 如果的小型的独立项目，那无需模块化。
  • 如果是大型的独立项目，那么建议根据功能或者类型来做模块化（SOA）。
  • 如果走在了时代的潮流中，选用了微服务的话，建议使用bom管理所有微服务的基本依赖以确保版本统一，而每个具体的微服务也至少抽成接口和具体的服务（根据具体的情况可以再抽）。
  • 如果项目是内嵌到别的项目里面去的框架或者组件，那么建议将整个项目彻底模块化，比如配置，接口，实现，依赖，组件，插件等都可以转化为独立的模块。一来根据灵活，方便做拓展，二来也方便客户重写。

关于分层

接入层（Controller、Action或者微服务里面的接口层），只负责转发和记录请求

服务层（Service），负责具体的逻辑实现

持久化层（Dao，Mapper等），负责数据的持久化

PS：如果有能力的话可以考虑DDD，分成4层，充血模型

  除了最上面的三层之后，还可能会抽出以下几层：

  • 拦截器层（位于接入层之上，拦截请求统一实现某些功能，不管是MVC还是微服务都可能会有）
  • 组件层（提供通用的组件服务，用于服务层调用）
  • 缓存层（提供常用数据的缓存，用于服务层调用）
  • 消息中间件层（提供消息的处理和推送，处理时调用服务层，被服务层调用推送消息）

  分层的意义是使得整个项目结构更加合理，减少冗余，提高可读性和健壮性，也更利于之后拓展和维护。当然如果是一个很小很小的项目的话，分层意义也不是很大，但是如果是要长期维护的话，建议考虑好分层。


关于ORM

  ORM框架这方面国内之前主要还是Mybatis和Hibernate的天下。如果习惯了Mybatis且要写复杂SQL，强烈建议使用tkMybatis或者MyBatis-Plus这种增强框架，谁用谁知道，如果是Hibernate的话也可以考虑转Spring Data Jpa(Spring全家桶不香吗)或者JOOQ这种轻量级的。

逆向工程

  如果是快速开发可以试下逆向工程，感觉还是挺方便的，根据数据库的表直接生成一整套代码。

连接池

  说到ORM，也必然会扯到连接池，抛去老旧的c3p0和dbcp，似乎只剩下了Druid和HikariCP，Druid胜在监控（有单独的页面），HikariCP作为SpringBoot内置的则是性能无敌。也可以根据自己的情况选择。

分页

  MyBatis-Plus功能强大，有很多插件，也自带了分页插件，tkMybatis的话可以外挂PageHelper做分页。Jpa和JOOQ也都自带了分页。

拦截器

  针对某些情况，增加拦截器是很好的选择，比如在多租户的情况下，可以统一对SQL进行处理，可以极大减少开发量。


关于缓存

  一般来说，稍微大型的项目里面或者使用人数稍多的项目都需要上缓存，毕竟数据库的瓶颈摆在那里，区别只是用几重缓存而已。常用的缓存主要分为两种：本地缓存的和分布式用的NoSQL，本地缓存的性能肯定是最好的，但是通常来说满足不了分布式的需求，但是在某些情况下依旧适用（比如某些不大变动的数据）。NoSQL里面常用的有Redis，MongoDB和Couchbase。Redis主要把数据存储在内存中，其”缓存”的性质远大于其“数据存储“的性质，其中数据的增删改查也只是像变量操作一样简单；MongoDB更像是一个“存储数据”的系统，增删改查可以添加很多条件，就像SQL数据库一样灵活，适用于海量数据的访问；Couchbase号称是Best NoSQL Cloud Database Service，由于个人没有用过，跳过，有兴趣的可能自己研究下。

缓存穿透

  缓存穿透是指缓存和数据库中都没有的数据，而用户不断发起请求，导致数据库压力过大。一般解决的办法有增加校验拦截这种请求和直接缓存null，复杂一点的有布隆过滤器。

缓存击穿

  缓存击穿指的是某个热点key失效的瞬间，大量请求导致数据库压力过大。一般解决的办法是加锁(分布式锁)，防止都去数据库重复取数据。

缓存雪崩

  缓存雪崩指的是缓存大批量到期，导致的数据库压力过大，就像雪崩一样。一般解决的办法有设置不同时间过期和或者部分设置为不过期。

保证最终一致性

  在并发的情况下，缓存可能会和数据库的值产生不一致的情况。主流解决方案是先更新数据库然后删除缓存的策略（延时双删）。


关于消息中间件

  消息中间件其实包括很多，4大MQ，我其实就接触过RabbitMQ和Kafka，ActiveMQ和阿里的RocketMQ则还没时间摸索。在项目需要和别的系统交互的时候，采用MQ无疑是最好的选择：支持异步，高并发，解耦，削峰，易拓展等等。
  个人观感里面，不同的MQ虽然核心逻辑差不多，但是实际的实现和根据其特点的应用场景都不一样。比如RabbitMQ是Push模式，服务端主动发送消息到客户端，及时性好；Kafka则是Pull模式，客户端主动从服务器端获取信息，实时较差但是针对性强。所以RabbitMQ适合企业级的消息发送订阅，而Kafka由于其高性能，常用日志采集，数据采集上。

确保消费成功和解决重复消费

  总的来说，MQ都会有自己的事务的机制或者确认的机制确保消息不丢失，一定消费成功，但是需要项目自己解决重复消费的问题（毕竟一旦消费时间就会重新消费）。解决的方法也有很多种，根据场景的不同增加额外的条件过滤即可。在使用MQ的时候，这两块是一定要注意的地方。


关于分布式事务

  关于分布式事务，有多种解决方案，两阶段提交，三阶段提交，补偿事务TCC，基于消息队列和 Sagas事务模型。不同的解决方案有不同的框架，当然也可以自己实现。由于这块涉猎也不是很多，之后看官可以自己研究。需要拎出来讲的原因是如果是分布式系统的话，这一块很重要……

关于国际化

  这一块其实也谈不上很有经验，但是以我的粗浅之见，解决策略是用code或者key来表示对应的文本信息，然后根据语言映射到不同的文件。前端和后台对于i18n其实都有很成熟的解决方案。需要拎出来讲的原因是一定要考虑清楚之后是不是会国际化，如果没有考虑的话，之后的返工会是很大的工作量，尤其是当文本信息是代码里面写死的时候。


关于异常处理

  • 个人认为，最好的策略是框架统一封装处理。
  • 需要处理异常的时候才捕捉异常，否则直接往上抛，直到有人处理或者被框架处理。
  • 如果有需要，可以对异常进行封装和分类，根据类型做不同的处理。

关于日志

  • 使用Java日志门面（如Slf4j），方便后期切换日志框架（如log4j2和logback）。
  • 合理规划级别，除了正常的info和error级别外，可以适量加下debug级别的日志，方便后期遇到问题排查。
  • 根据功能和类型的不同，可以输出到不同的日志文件。
  • 如果对实时性没有要求，尽量使用异步日志模式而不是同步日志模式。
  • 历史的日志文件建议根据年月日压缩归档。

关于接口和文档

  • 接口风格尽量保持统一，比如用了RESTful就全部都用REST风格，约定了接口都以get，pus，post和delete开头就以这些开头。
  • 如果的mvc的项目，强烈建议使用Swagger2（据说微服务也支持），不仅方便形成接口文档，还可以直接调用，利于测试和开发。
  • 或者可以使用apiDoc生成形成接口文档，样式简洁，美观大方（至少比javadoc好看很多），缺点是要写的注解比较多。
  • 如果不介意在外部维护接口文档的话，也很推荐使用yApi，缺点是需要额外部署服务器，但是优点是功能强大，更加适合于项目开发。

关于编码规范

  开发人员建议统一使用代码规范插件，常用的插件有：SonarQube，Fortify，阿里代码规范插件。用起来感觉都不错，可以约定一套习惯给项目配置规则。Sonar还可以与Jenkins集成，在CI集成的时候扫描，但是终究还是在源头上控制更有效，只能作为保险。


关于单元测试和自动化测试

  单元测试最好还是要有的，从顶层开始写用例可以尽量覆盖到全部代码。如果没有单元测试的话，可以寄希望于自动化的接口测试，通过自动化测试把控代码质量，否则很可能哪天接口就被改坏了，造成了严重的事故。自动化测试的工具也有很多，如果有能力甚至可能根据项目的情况自研。


关于DevOps

  DevOps可以说是现在的潮流，CI/CD的自动化和白屏化会给工程人员极大的便利和解放，也间接增强了项目的稳定性。Jenkins作为最受欢迎的自动化工具是肯定是我们的首选，里面有丰富的插件和扩展，基本可以满足所有的需求。


关于云和Docker

  个人认为，上不上云还是看需求的，对于一些小公司和个人项目，上云是绝对有必要的…Docker这个很有说法， 我个人强烈建议新的项目部署的时候做成Docker镜像的形式的部署，虽然在某些情况下，可能是得不偿失的..毕竟变相增加了难度，但是为了未来考虑，镜像化是一个很好的选择。

StringUtils的使用

2020-04-29
编程

  这篇文章是我对之前学的StringUtils的整理，没什么技术含量，适用于刚刚毕业的崽，如果你已经是个中老手，就不用浪费时间往下看了。

介绍

  StringUtils是一个apache基金会提供的用于字符串处理的常用工具类，功能强大，使用方便，到几乎每个项目里面都用了，每个Java开发都或多或少见过和用过。熟练地掌握并使用它会让你在开发中事半功倍，代码可读性更高且更加健壮。

依赖


空字符串

  很多时候，我们在判断一个字符串是不是空字符串的时候，往往需要考虑很多情况，比如字符串对象是不是null，是不是空，是不是空格等。这会导致我们的判断写的很长，且容易遗漏某些判断。所以强烈建议使用StringUtils里面提供的对空字符串的判断的方法，简单明了。这些方法可以判断某一个字符串，也可以批量判断。常用的是判断某一个字符串的是不是空字符串的有以下4个方法：
1、public static boolean isEmpty(String str)：判断某字符串是否为空，为空的标准是为null或者长度为0

2、public static boolean isNotEmpty(String str)：判断某字符串是否不为空，不为空的标准是不为null且长度大于0（就是isEmpty()方法的反向判断）

3、public static boolean isBlank(String str)：判断某字符串是否为空，为空的标准是为null或者长度为0或者全是空白符组成

4、public static boolean isNotBlank(String str)：判断某字符串是否不为空，不为空的标准是不为null且长度大于0且不全是空白符组成（就是isBlank()方法的反向判断）

  我们可以发现isEmpty()和isBlank()不一样的地方是isBlank()更加严格，连空格等都算成是空的。个人感觉isBlank()和isNotBlank()会比isEmpty()和isNotEmpty()更建议用。

大小写

  Java原生的String类中就有一些对大小写的处理，与原生的不用的是，StringUtils兼容了对null的处理，当被处理的字符串为null时，会返回null（避免了空指针的报错）。

  StringUtils而且还拓展了更多的方法，首字母大写或者小写，判断字符串是不是全大写或者小写。

查找

  String原生也支持了indexOf的查找，不同的是，StringUtils兼容了对null的处理，除此之外还可以支持查询第几次出现的位置。

  • public static int indexOf(String str, char searchChar, int startPos)：返回字符 searchChar 从 startPos 开始在字符串 str 中第一次出现的位置。
  • public static int ordinalIndexOf(String str, String searchStr, int ordinal)：返回字符串 searchStr 在字符串 str 中第 ordinal 次出现的位置。

替换

  同样的，StringUtils的replace也兼容了对null的处理，除此之外还支持只会替换一次或者指定最大替换的个数。

移除

  还是同样的，StringUtils的remove也兼容了对null的处理，除此之外支持移除开头和结尾的（其实我们也可以用replace成空的字符串来完成remove的操作，但是个人不是很建议）。

截取

  截取也是我们常用的方法，StringUtils的截取支持指定位置，或者指定字符串前后截取。

反转

  反转是一个很有意思的功能，StringBuffer也有reverse方法，但是如果想直接处理还需要把String转为StringBuffer就比较麻烦，还是StringUtils的reverse比较简单。

  StringUtils的reverse还支持根据字符镜像反转。

其他

  commons-lang3包中， 除了StringUtils这个常用类之后，还有ArrayUtils（数组工具类），CharUtils（字符串工具类）,NumberUtils(数字工具类)等，大家有兴趣也可以去了解下！
